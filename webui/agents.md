## agents.md – Техническое задание для разработки Web UI инструмента «DekatronPC Block Place & Route»

Данный документ предназначен для AI-агентов (разработчиков), которые будут создавать инструмент поэтапно. Он содержит полное описание функциональности, архитектуры, моделей данных и алгоритмов.

---

### 1. Общее описание проекта

**Цель:**  
Разработать веб-приложение (SPA с возможным бэкендом для алгоритмов) для интерактивного размещения и трассировки ламповых логических блоков компьютера DekatronPC. Инструмент получает на вход структурный Verilog-нетлист, liberty-файл и описания аппаратных модулей (KiCad netlist). Пользователь вручную и автоматически размещает элементы по модулям, модули в рядах блока, затем трассирует межсоединения ортогональными проводами. Результат позволяет вести физический монтаж: провода отображаются на схеме, можно помечать выполненные сегменты и экспортировать таблицу соединений.

**Ключевые характеристики:**
- Блок: 3–7 горизонтальных рядов, до 24 модулей в ряду (может варьироваться).
- Модуль: печатная плата с контактным разъёмом 2×36 (пины 1A–36A слева, 1B–36B справа). Ширина модуля кратна 12 мм (обычно 24, 36, 96 мм; блоки питания до 100 мм). На модуле размещаются логические элементы (из liberty) или составные блоки (например, декатрон).
- Провода: изолированные, один слой, пересечения разрешены. Трассировка ортогональная, сегменты идут строго по границам модулей (каналам) и по периметру блока.

---

### 2. Стадии рабочего процесса (вкладки UI)

Интерфейс построен вокруг вкладок, соответствующих последовательности проектирования:

1. **Project** – создание/открытие проекта, общие настройки.
2. **Netlist** – загрузка Verilog-нетлиста и liberty-файла, просмотр списка цепей и экземпляров логических элементов.
3. **Elements** – ручное создание/редактирование элементов, отсутствующих в liberty (сложные модули, блоки питания), с определением пинов.
4. **Modules** – создание и конфигурация модулей: загрузка KiCad netlist, сопоставление выводов, задание типов логических ячеек и их количества, ширина модуля.
5. **Placement** – интерактивное размещение модулей в блоке и элементов внутри модулей. Автоматическая оптимизация, ручное перетаскивание, фиксация.
6. **Routing** – трассировка соединений ортогональными проводами (автоматическая и ручная), настройка цветов, маркировка выполненных цепей.
7. **Assembly** – просмотр окончательной схемы, экспорт таблицы соединений, фильтрация по готовности.

В любой момент можно переключаться между вкладками; изменения сохраняются в общем состоянии проекта.

**Мульти-блочная архитектура:** Проект поддерживает 2-3 вычислительных блока (IpLine, ApLine, MachineCtrl),
каждый со своим нетлистом, размещением и трассировкой. Liberty-файл и аппаратные модули — общие.
Блоки переключаются через селектор на вкладке Netlist. Имя блока — из имени файла Verilog.

### 3. Функциональные требования

#### 3.1. Управление проектом
- Хранение проекта в формате JSON (собственная структура). Возможность загрузки/сохранения локально или на сервер.
- История изменений (undo/redo) глобально.
- Автосохранение или ручное сохранение.

#### 3.2. Исходные данные
- **Verilog-нетлист**: плоский структурный, содержит объявления wire и экземпляры примитивов (имена соответствуют liberty-ячейкам либо особым модулям). Пример: `AND2 U1 (.A(net1), .B(net2), .Y(net3));`
  - **Мульти-блок**: можно загрузить 2-3 нетлиста. Каждый создаёт отдельный блок в проекте.
    Имя блока — из имени файла (`IpLine_synth.v` → `IpLine`).
- **Liberty-файл**: стандартный Synopsys `.lib`, из которого извлекаются пины и функциональное описание. Для простоты можно ограничиться именем ячейки и направлением пинов. Более сложные временные параметры не требуются.
  - Liberty-файл ОДИН на весь проект — общий для всех блоков.
- **Пользовательские элементы**: создаются вручную на вкладке Elements (имя, список пинов: вход/выход/питание).
- **Модули (аппаратные платы)**: конфигурация включает:
  - Ширина (в шагах 12 мм или в мм).
  - Описание разъёма 2×36 (физические контакты A1..A36, B1..B36).
  - Список логических слотов: каждый слот имеет тип (привязка к liberty-ячейке или пользовательскому элементу), количество экземпляров этого типа, и список выводов слота, сопоставленных с контактами разъёма. Сопоставление происходит через KiCad netlist модуля (иерархический лист схемы) – пользователь загружает netlist, выбирает компонент на схеме, его пины автоматически матчатся с контактами по именам цепей. Ручная корректировка возможна.
  - Доп. свойства: потребляемая мощность, тепловыделение (для информации).

#### 3.3. Блок и компоновка
- Параметры блока: количество рядов (3–7), максимальное количество позиций в ряду, вертикальный шаг (расстояние между центрами модулей), отступы от краёв, ширина корпуса. Все размеры в мм.
- Модули могут занимать несколько «шагов» по горизонтали (ширина 24 мм = 2 слота, 36 мм = 3 слота, 96 мм = 8 слотов и т.д.). Сетка 12 мм.
- Запретные зоны: многоугольники/прямоугольники, задаваемые пользователем на поле блока (DXF импорт или ручное рисование). В этих зонах трассировка запрещена.

#### 3.4. Размещение (Place)
- **Начальное состояние:** после загрузки всех данных элементы нетлиста не привязаны к модулям, модули не расставлены в блоке. Пользователь видит пустой блок и перечень модулей/элементов сбоку.
- **Автоматическое распределение элементов по модулям:**
  - Каждый модуль имеет известный список слотов (тип и количество). Элементы из нетлиста сопоставляются по типу со слотами. Один экземпляр элемента занимает один слот соответствующего типа.
  - Цель: разместить все экземпляры так, чтобы суммарная длина связей (HPWL) была минимальной. Клоковые сети учитываются отдельно: регистры одного домена стремятся к компактному размещению вблизи источника клока.
  - Ограничения: ручная фиксация (locked) элемента/модуля. Зафиксированные объекты не перемещаются автоматически.
- **Интерактивная ручная корректировка:**
  - Перетаскивание элемента внутри модуля (между слотами) или в другой модуль. Если в целевом слоте уже есть незаблокированный элемент, он сдвигается в ближайший подходящий свободный слот (в этом же модуле или в другом, согласно правилам). Заблокированные элементы остаются на месте.
  - Перетаскивание целого модуля в другую позицию ряда или в другой ряд. Все незаблокированные модули сдвигаются, освобождая место (как в адаптивных интерфейсах).
  - Блокировка/разблокировка элемента или модуля (замочек).
- **Метрика качества размещения:** суммарная длина полупериметров (HPWL) для всех цепей. Веса для клоковых цепей могут быть повыше. Обновляется в реальном времени.
- **Визуализация:** на поле блока модули – вертикальные прямоугольники с контактами. Внутри каждого модуля можно показать/скрыть слоты и назначенные элементы (галочка слоя). При клике на элемент подсвечиваются связанные пины и принадлежащие цепи.

#### 3.5. Трассировка (Route)
- **Геометрия трасс:** ортогональные ломаные линии, состоящие из горизонтальных и вертикальных сегментов. Каждая цепь (net) соединяет два или более пинов модулей. Провод не имеет ответвлений: из одного пина выходит одна линия, которая идёт к другому пину; если цепь объединяет несколько пинов, то от одного пина идёт путь к другому, затем к следующему – формируется маршрут «точка-точка»? Согласно уточнению, «провод между двумя модулями — это целое соединение между двумя пинами модулей. Провод прокладывается единым куском…» Значит, многотерминальная цепь будет представляться как множество двухтерминальных соединений, каждое из которых трассируется отдельно? Но пользователь ожидает видеть сеть как единое целое. Логично для каждой цепи строить дерево Штейнера (степень 2 и более), но физически провод могут прокладывать от одного пина к другому последовательно, разрезая при необходимости. Упростим: на этапе трассировки мы автоматически создаём множество двухпиновых сегментов, соединяющих все пины цепи. Пользователь может вручную добавить/удалить сегменты.
- **Каналы трассировки:** линии прокладываются строго по промежуткам между модулями (горизонтальные и вертикальные дорожки). Ширина канала равна зазору между модулями. Провода могут идти по периметру блока (внешняя граница). Запретные зоны блокируют каналы.
- **Автоматическая трассировка:** для каждой двухтерминальной связи выполняется поиск пути по ортогональной сетке каналов. Алгоритм A* или волновой (Lee) с учетом только запретных зон. Пересечения с другими проводами разрешены и не штрафуются. Допускается избыточность (пути не оптимизируются по длине, т.к. длина уже минимизирована размещением).
- **Ручная трассировка:** инструмент «карандаш»: кликом на пин модуля начинаем вести линию, каждый клик ставит точку поворота (только ортогонально), завершается на другом пине. Можно редактировать существующие маршруты, перетаскивая сегменты.
- **Визуализация:**
  - Провода отображаются цветными линиями заданной толщины (2–4 пикселя). Цвет назначается вручную или автоматически по палитре на основе имени цепи.
  - При наведении подсвечивается вся цепь, остальные элементы затемняются.
  - Клик по сегменту выделяет всю связанную двухпиновую связь (цепь), показывая её полностью. Режим «Assembly» позволяет пометить эту связь как выполненную (становится, например, зелёной и полупрозрачной или с галочкой).
- **Учёт выполненных соединений:** в состоянии проекта для каждой цепи хранится флаг `routed` и `assembled`. На вкладке сборки можно фильтровать только готовые или только неназначенные.

#### 3.6. Экспорт
- Генерация таблицы соединений (CSV/JSON): список цепей, для каждой – перечень пинов и координат сегментов (трасс) с указанием, выполнена ли цепь.
- Экспорт схемы как изображения (PNG/SVG).

### 5. Архитектура приложения

#### 5.1. Общий стек (предложение)
- **Frontend:** React + TypeScript, управление состоянием через Redux Toolkit или Zustand. Визуализация: HTML5 Canvas (например, Konva.js) для производительности при большом количестве объектов.
- **Backend (опционально):** Node.js/Python для запуска алгоритмов размещения и трассировки. Для первого этапа можно реализовать на клиенте с Web Workers, чтобы не усложнять деплой. Сервер потребуется, если проект станет многопользовательским или нужна централизованная БД.
- **Алгоритмические библиотеки:** для размещения – модифицированный симулятор отжига или min-cut (можно реализовать самостоятельно). Для трассировки – поиск пути на сетке (A*).
- **Хранение:** локальный файл JSON, загружаемый/сохраняемый через браузер. Для серверного варианта – файловая система или PostgreSQL.

#### 5.2. Компонентная структура (основные блоки)
- `App` – контейнер с вкладками.
- `ProjectManager` – загрузка/сохранение JSON.
- `LibertyView` – парсинг и отображение ячеек.
- `ElementEditor` – создание/редактирование пользовательских элементов.
- `ModuleEditor` – загрузка KiCad netlist, маппинг слотов, задание ширины.
- `BlockCanvas` – основное полотно для размещения и трассировки. Реагирует на масштаб и панорамирование. Имеет несколько слоёв:
  - Слой запретных зон и границ блока.
  - Слой модулей (прямоугольники с пинами).
  - Слой внутренностей модуля (опционально).
  - Слой связей (линии предварительных соединений, «воздушные» провода при выборе цепи).
  - Слой трассировки (провода).
- `PlacementTool` – логика перетаскивания, блокировки, вызова авторазмещения.
- `RoutingTool` – логика ручной и автоматической трассировки, выбора цвета, пометки сборки.
- `ExportPanel` – генерация таблицы, экспорт изображений.

#### 5.3. Сервисы и алгоритмы (место выполнения)
- **Auto-placer:**
  - Первый этап – распределение элементов по слотам модулей: решается как задача назначения (assignment) с ограничениями ёмкости. Можно использовать венгерский алгоритм с последующей итеративной оптимизацией обменов.
  - Второй этап – размещение модулей в блоке: симуляция отжига или min-cut. Целевая функция: сумма HPWL всех цепей (центры модулей), плюс штрафы за выход за границы ряда, нарушение фиксации. Клоковые связи: умножаются на вес >1, чтобы подтягивать регистры ближе к источнику.
- **Auto-router:** поиск пути A* на графе каналов. Узлы – пересечения вертикальных и горизонтальных каналов. Вес ребра – его длина. Запретные зоны удаляют недоступные ребра. Для многотерминальных цепей строим минимальное остовное дерево на графе, затем каждое ребро маршрутизируем независимо.

### 6. Взаимодействие с KiCad netlist (модули)

На вкладке Modules пользователь:
1. Загружает файл `.net` (KiCad netlist). Приложение парсит иерархические ссылки.
2. Выбирает корневой символ модуля, затем суб-листы с логическими элементами.
3. Программа автоматически сопоставляет имена цепей с пинами выбранных компонентов и контактами разъёма (по совпадению имён). Результат отображается в таблице, можно скорректировать вручную.
4. Определяется ширина модуля (число шагов 12 мм).

### 7. UI/UX детали

- **Масштабирование и панорамирование:** колёсико мыши для зума, зажатие средней кнопки или пробел+перетаскивание для перемещения. Ограничение мин/макс зума.
- **Слои отображения:** панель с чекбоксами: «Подписи», «Элементы внутри модулей», «Трассы», «Запретные зоны», «Выполненные трассы».
- **Перетаскивание:** при захвате элемента (внутри модуля) или модуля подсвечиваются допустимые целевые области. Если место занято, соседние объекты анимированно сдвигаются.
- **Фиксация:** иконка замка на элементе/модуле, клик переключает.
- **Подсветка цепей:** клик на проводе или в списке цепей (должен быть список слева) делает цепь яркой, всё остальное – серым с прозрачностью.
- **Цвета:** палитра из 20 различных цветов, назначается вручную через панель свойств цепи.

### 8. Поэтапный план реализации

1. **Фундамент:**
   - Инициализация проекта (React + Canvas/Konva). Структура папок. Менеджер состояния.
   - Парсер Verilog-нетлиста (простые регулярки) и liberty-файла (JSON после парсинга).
   - Загрузка/сохранение проекта.
2. **Редактор элементов и модулей:**
   - Интерфейс создания пользовательских элементов.
   - Парсер KiCad netlist (упрощённый: только список компонентов и цепей без иерархии). Маппинг пинов.
3. **Блок и размещение модулей:**
   - Визуализация блока с рядами. Ручное добавление модулей на поле, перетаскивание, фиксация.
   - Автоматический placer (базовый).
4. **Размещение элементов внутри модулей:**
   - Интерфейс слотов, перетаскивание элементов.
   - Интеграция с общим placer'ом.
5. **Трассировка:**
   - Построение графа каналов по координатам модулей и препятствий.
   - Автоматическая трассировка двухпиновых цепей.
   - Ручной инструмент.
6. **Управление цепями и сборка:**
   - Назначение цветов, подсветка, отметка выполненных цепей.
   - Экспорт таблицы соединений.
7. **Полировка и тестирование:**
   - Undo/redo, производительность для 200+ модулей и 1000+ трасс.

### 9. Открытые вопросы и возможные упрощения

- **Клоковое дерево:** конкретные правила размещения регистров пока не формализованы. На первом этапе можно добавить возможность назначать вес цепям вручную, что повлияет на placer.
- **Ёмкость каналов и пучки проводов:** для простоты игнорируем пересечения и толщину. Если провода будут налезать друг на друга визуально, можно добавить небольшое разнесение.
- **Сервер или клиент:** рекомендуем начать с полностью клиентского решения и Web Workers для алгоритмов, оставив возможность позже вынести вычисления на сервер.
- **Поддержка DXF:** сложный функционал, можно отложить, ограничившись рисованием запретных зон вручную.
