# Комплексный план разработки и сборки лампового компьютера DekatronPC

## Текущее состояние (июль 2026)

### Что уже есть
| Компонент | Состояние |
|-----------|-----------|
| **RTL (SystemVerilog)** | Полное описание компьютера. Работает на FPGA Altera, частично верифицирован. Программы Brainfuck исполняются корректно. Требует оптимизаций и рефакторинга. Возможен неполный охват фич. |
| **FPGA-стенд** | Плата Altera + дисплей + кнопки. Работает на целевой частоте. Предназначен для эмуляции несобранных блоков. Платы интерфейса отсутствуют. |
| **Структурный нетлист** | Получен после синтеза под ламповую библиотеку. Требует обновления после доработки RTL и liberty. |
| **Liberty-файл** | Предварительный список из 11 ячеек (BUF_N16, NOT_N16, OR2_N16, OR4_N16, NOR2_N16, NOR4_N16, AOI21_J2N16, AOI22_J2, NAND_J2, LATCH_J2N16, DFF_J2N16). Временные характеристики не прописаны. |
| **Модули (механика)** | Типы и размеры предварительно определены. Эскиз модуля имеется. Известна раскладка контактов питания. Количество разных типов модулей — неизвестно. KiCad-проектов нет. |
| **Испытательный стенд** | В стадии проектирования. 3 разъёма под модули, источники анодного/смещения/накала с индикацией, генератор сигналов, байонеты для приборов. |
| **Первый модуль** | D-триггер (DFF_J2N16) — в разработке. |
| **WebUI Place & Route** | Этапы 0 и 1 завершены: каркас React-приложения, типы данных, Zustand-стор с undo/redo, парсеры Verilog и Liberty, редактор элементов. |

## Карта рабочих направлений (workstreams)

Работы ведутся по шести параллельным направлениям. Стрелки показывают ключевые зависимости.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    WORKSTREAM A: RTL & Synthesis                      │
│  RTL-оптимизация ──► Верификация ──► Полный синтез под liberty       │
│                            │                        │                 │
│                            ▼                        ▼                 │
│                     WORKSTREAM D:          WORKSTREAM E:              │
│                  Test Bench                 WebUI Tool                │
│                        │                        │                     │
│                        ▼                        │                     │
│               WORKSTREAM B:                     │                     │
│           Liberty & Cell Characterization       │                     │
│                        │                        │                     │
│                        ▼                        ▼                     │
│               WORKSTREAM C:            WORKSTREAM F:                  │
│           Module PCB Design          Block Assembly                  │
│          (KiCad схемы + платы)      (физический монтаж)              │
└─────────────────────────────────────────────────────────────────────┘
```

## Workstream A: RTL и синтез

**Цель:** получить верифицированное RTL-описание, готовое к синтезу под целевую ламповую библиотеку, и итоговый структурный нетлист.

**Владелец:** разработчик RTL.

### A.1 Ревизия и рефакторинг RTL

| # | Задача | Детали |
|---|--------|--------|
| A.1.1 | Провести аудит текущего RTL | Составить карту модулей: IpLine, ApLine, MachineCtrl, счётчики, память, декодер инструкций. Отметить реализованные и отсутствующие фичи. |
| A.1.2 | Рефакторинг под синтез | Убрать несинтезируемые конструкции (`initial`, `$display`, временные задержки). Привести к единому стилю. |
| A.1.3 | Оптимизация FSM | Упростить конечные автоматы (см. wiki: FSM IpLine, MachineCtrl). Минимизировать количество состояний, убрать избыточные переходы. |
| A.1.4 | Документирование интерфейсов | Для каждого модуля зафиксировать порты и протокол Request/Ready. |
| A.1.5 | Полная верификация на FPGA | Прогнать тестовый набор brainfuck-программ, убедиться в корректности всех инструкций (`+-<>.,[]`, debug ISA). |

**Результат:** чистый, верифицированный RTL, готовый к синтезу.

### A.2 Синтез под ламповую библиотеку

| # | Задача | Детали |
|---|--------|--------|
| A.2.1 | Обновить liberty до актуального состояния | Синхронизировать с Workstream B. |
| A.2.2 | Запустить синтез Yosys: `read_verilog → synth → abc -liberty` | Получить плоский структурный нетлист. |
| A.2.3 | Проанализировать статистику синтеза | Количество экземпляров каждого типа ячеек, общее количество ламп, критический путь. |
| A.2.4 | Итеративно улучшать | Если синтез даёт неоптимальный результат — корректировать RTL или liberty. |
| A.2.5 | Разделить нетлист по блокам | IpLine, ApLine, MachineCtrl — три отдельных нетлиста для трёх 4U-блоков. |
| A.2.6 | Оценить количество модулей | Исходя из нетлиста — сколько плат каждого типа потребуется. |

**Результат:** структурный нетлист, разбитый по блокам, с известным количеством ячеек каждого типа.

## Workstream B: Liberty и характеризация ячеек

**Цель:** получить точные временные и энергетические характеристики для каждого лампового примитива.

**Владелец:** разработчик схемотехники.

### B.1 Предварительная liberty (уже есть)

Текущий список ячеек:

| Ячейка | Назначение | Лампы |
|--------|-----------|-------|
| `BUF_N16` | Буфер (триод 6Н16Б) | 1×6Н16Б (половинка) |
| `NOT_N16` | Инвертор (триод 6Н16Б) | 1×6Н16Б (половинка) |
| `OR2_N16` | 2-входовое ИЛИ | 2×6Н16Б |
| `OR4_N16` | 4-входовое ИЛИ | 4×6Н16Б |
| `NOR2_N16` | 2-входовое ИЛИ-НЕ | 2×6Н16Б |
| `NOR4_N16` | 4-входовое ИЛИ-НЕ | 4×6Н16Б |
| `AOI21_J2N16` | 2-1-AND-OR-Invert (пентод 6Ж2Б + триод) | 1×6Ж2Б + 1×6Н16Б |
| `AOI22_J2` | 2-2-AND-OR-Invert (пентод 6Ж2Б) | 1×6Ж2Б |
| `NAND_J2` | И-НЕ (пентод 6Ж2Б) | 1×6Ж2Б |
| `LATCH_J2N16` | Защёлка | 1×6Ж2Б + 1×6Н16Б |
| `DFF_J2N16` | D-триггер | 1×6Ж2Б + 1×6Н16Б |

### B.2 Характеризация

| # | Задача | Детали |
|---|--------|--------|
| B.2.1 | Разработать принципиальные схемы каждого примитива | В KiCad: транзисторный уровень на лампах, номиналы резисторов, напряжения. |
| B.2.2 | Запустить SPICE-симуляцию (ngspice) | Из KiCad экспортировать нетлист, прогнать переходные процессы. |
| B.2.3 | Снять задержки `rise_delay`, `fall_delay` | Для каждого пина → выхода. Записать в liberty. |
| B.2.4 | Снять энергопотребление | Ток анода, ток накала в статике и при переключении. |
| B.2.5 | **Физическая верификация на стенде** | Собрать тестовую плату с одним примитивом, подать сигналы с генератора, снять осциллограммы (используя Workstream D). |
| B.2.6 | Уточнить liberty по результатам измерений | Скорректировать задержки, пороги, нагрузочную способность. |
| B.2.7 | Автоматизировать генерацию liberty из KiCad | Скрипт, извлекающий параметры из симуляции и формирующий liberty-строки. |

**Результат:** выверенный liberty-файл с реальными (не «с потолка») характеристиками.

## Workstream C: Проектирование модулей (KiCad)

**Цель:** создать библиотеку печатных плат модулей — принципиальные схемы и топологию — покрывающую все типы ячеек из liberty.

**Владелец:** разработчик схемотехники / трассировщик PCB.

### C.1 Подготовка

| # | Задача | Детали |
|---|--------|--------|
| C.1.1 | Создать KiCad-библиотеку символа разъёма | ГРПМШ 2×36: контакты A1..A36 (лево), B1..B36 (право). Один ряд ключ/питание. |
| C.1.2 | Создать KiCad-библиотеку ламп | 6Н16Б, 6Ж2Б, 6Х7Б, А110 — символы и посадочные места. |
| C.1.3 | Создать шаблон модуля | Плата 140×140 мм, крепёжные отверстия, зона разъёма. |

### C.2 Разработка модулей — порядок по приоритету

| C.2.6 | **Модуль памяти** (ферритовые кубы?) | TBD | TBD | Низкий |

### C.3 На каждый модуль

## Ключевые риски и mitigation

| Риск | Вероятность | Mitigation |
|------|------------|------------|
| RTL содержит недокументированные баги | Средняя | Полная верификация на FPGA до начала синтеза (A.1.5) |
| Схемотехника ламповых ячеек неработоспособна на целевой частоте | Средняя | Ранний физический прототип DFF (C.2.1) + измерения на стенде (D) |
| Задержки ламповых ячеек делают целевую частоту недостижимой | Средняя | Итеративный цикл RTL ↔ Liberty ↔ симуляция. Если частота падает — пересмотр архитектуры конвейера |
| Количество ламп превышает бюджет | Низкая | Синтез даёт точную цифру. При превышении — оптимизация RTL |
| Тепловыделение 5 кВт сложно отвести | Средняя | Расчёт airflow на раннем этапе. Тест с одним блоком до сборки всех трёх |
| KiCad-схемы модулей задерживаются | Высокая | Режим заглушек в WebUI (E.5) позволяет вести размещение и трассировку без точных схем |

## Открытые вопросы для принятия решений

1. **Ферритовая память:** какой тип ОЗУ используется в итоговой машине? Влияет на типы модулей и интерфейсы.
2. **Межблочные соединения:** три 4U-блока соединяются кабелями? Какой разъём? Нужно отразить в модели блока.
3. **Уровни сигналов:** анодное +150V — это уровни логики? Или после усилителей сигналы приводятся к другому уровню? Влияет на liberty (логические пороги).
4. **KiCad-симуляция → liberty:** какой формат автоматической генерации? Нужен скрипт-конвертер из SPICE-отчёта в liberty.
5. **FPGA как эмулятор несобранных блоков:** какой интерфейс между FPGA-платой и ламповыми блоками? Как происходит «замена» — поканально?
